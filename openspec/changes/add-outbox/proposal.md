## Why

Projections and (later) external consumers must learn about domain events **reliably** — no
lost events, no silent double-effects — without coupling the write path to those consumers.
The answer is a **transactional outbox**: events are durably recorded in the same transaction
as the aggregate append, and a separate **relay worker** publishes them, in order, to a
transport. Delivery is at-least-once; consumers are made idempotent so re-delivery has no
observable effect. The proof: killing the relay mid-batch and restarting loses no events and
double-delivers none observably.

## What Changes

- Treat the **event log as the outbox**: the event store already writes events in the
  aggregate's append transaction, so no dual-write is needed. The relay tails the global
  stream from a **published checkpoint** (ADR-002: outbox over dual-write/CDC).
- Introduce an **`EventPublisher`** transport port with adapters: an **in-memory** one (tests),
  and a **PostgreSQL `LISTEN/NOTIFY`** one for the demo. The port is shaped so **NATS JetStream
  (fiber-based `thesis/nats`)** can be swapped in later without touching the relay.
- Introduce an **`OutboxRelay`** that publishes each unpublished event in global order and
  advances the checkpoint **only after a successful publish** — so a crash never loses an event
  and at most re-publishes the in-flight one (at-least-once).
- Make consumers **idempotent by event id** via a `consumed_events` guard, so at-least-once
  delivery is effectively-once per consumer. (The projection runner is already idempotent by
  position.)
- Add an **`outbox:relay`** console worker (one pass or loop). The RoadRunner job host and a
  fiber-based relay loop arrive with deployment/observability; the relay logic is unchanged.
- Generalize the checkpoint table to a shared `stream_checkpoints` (schema configurator →
  generated migration).

## Capabilities

### New Capabilities
- `outbox`: reliable, ordered, at-least-once publication of domain events from the event log
  via a relay worker over a swappable transport, with crash-safe checkpointing and idempotent
  consumers.

## Impact

- **New code:** `App\Outbox\` — `EventPublisher` port, `InMemoryEventPublisher`,
  `PostgresNotifyEventPublisher`, the `OutboxRelay` + its checkpoint, a `ConsumedEvents`
  idempotency guard, and the `outbox:relay` command.
- **Database:** rename `projection_checkpoints` → `stream_checkpoints`; add `consumed_events`
  (consumer, event_id). Both via the schema configurator; migration generated by `diff`.
- **Depends on** `event-store` (global read). The relay reads the events log; transports are
  abstracted. No new third-party dependencies in this change (NATS is a later swap-in).
- **Downstream:** the projectors can be driven by the relay; `add-deployment` runs the relay as
  a RoadRunner worker; `add-observability` exports `outbox_pending`.
